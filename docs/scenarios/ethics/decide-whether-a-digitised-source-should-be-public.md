# How do I decide whether a digitised source should be public?

<div class="answer-meta" markdown>
<span>Ethics</span>
<span>intermediate</span>
<span>45–90 min</span>
</div>

## What you are trying to do

A source is technically digitizable, but publication may affect living people, communities, or cultural rights.

!!! quote "One-sentence version"
    Start with a tiny sample, keep provenance, and only scale up once the workflow survives testing.

## You need

- Source or dataset
- Stakeholders
- Intended audience

## Tools

- [Consent checklist](../../contribute/style-guide.md)

## Workflow

1. Identify who is represented, not only who owns the file.
2. Check legal rights, platform terms, and community expectations.
3. Separate access for teaching, research, and public reuse.
4. Minimize sensitive details where possible.
5. Document the decision and revisit it later.

## Output

Access decision: public, restricted, anonymized, delayed, or not published.

## Check yourself

- Could publication harm or expose someone?
- Would represented people recognize the use as fair?
- Are takedown/contact routes visible?

## Common traps

- Equating old with harmless.
- Assuming public record means ethical republication.
- Letting convenience decide access.

## Worked examples

- [source-grounded Slovenian language-advisor architecture](../../projects/jezikovni-svetovalec.md) is a useful cautionary architecture for source-grounded work where source texts may not be redistributable.
- [static knowledge graph, map, and corpus index example](../../projects/ladakh-relations.md) shows why public visibility, cultural context, and access decisions matter in heritage-oriented DH projects.
- [Slovenian comma-checking Word add-in example](../../projects/vejice-add-in.md) shows how application code can separate safe mock mode from remote processing of user text.

## Practice task

Run a mock ethics board: each group argues for public, restricted, or no publication.

## Useful extension

Turn the workflow into a small reusable checklist: input, command or tool action, output, quality check, and known failure mode.
